# 実行時の動き

goal-setter は、Goal の文章を作るスキルです。新しい実行の仕組みを追加するものではありません。Codex や Claude Code に元からある機能を使います。

## 発動のしかた

| 場面 | 動き |
| --- | --- |
| Codex で Goal の発動を依頼された | Codex の Goal 機能でそのまま設定する |
| Codex で下書きだけを依頼された、または Goal 機能がない | そのまま送れる `/goal ...` 行を返す |
| Claude Code | ユーザーが送る `/goal ...` 行を返す |

goal-setter は Goal を通常どおり設定します。並行探索、コンテキスト分離、独立検証が完了判断を改善する時は、実行中の Codex が対象の作業で subagent を起動し、その証拠を待ち、統合してから判断するという具体的な指示を Goal に入れます。Goal の受付自体は、依頼の理解に大きな独立調査が必要な場合を除き、親のコンテキストで行います。subagent のためだけに、ユーザーへ `/goal ...` を送り直してもらう必要はありません。人数や wave の形は、ユーザーが固定しない限り親が決めます。

`create_thread` は subagent ではなく、ユーザーから見える別の Codex タスクを作る機能です。ユーザーが別タスク、別スレッド、または worktree への分割を明示した場合だけ使います。

## 分けて進める作業

作業を分けるのは、次のような場合だけです。

- それぞれの成果が独立している
- それぞれを別々に確認できる
- 分ける手間より、短縮できる時間の方が大きい
- 最後にまとめる方法が見えている

分けられるかどうかは、ファイルの場所だけでは決めません。実際の動きがどれだけつながっているか、同じ状態を触るか、あとでまとめる時に壊れやすいかを先に見ます。ユーザーが数を指定していない限り、別の確認役の数は固定しません。親になる作業が、必要な数だけ呼び、結果をまとめてから次を判断します。

## 長い作業

長い自律作業では、Goal に「未確認一覧を更新しながら進む」動きを入れられます。実行する側は `execution-notes.md` に、残っている確認、見た証拠、合格・不合格・保留・詰まり、途中の判断を短く残します。確認するたびに状態を更新し、完了条件を満たしておらず、止まる条件にも当たらないなら、次に一番危ない未確認項目か、一番あいまいな未確認項目へ進みます。「次にやること」を書いただけでは止まりません。

## 深掘り確認

goal-setter が質問するのは、答えによって次のどれかが変わる時だけです。

- 完了条件
- 確認方法
- 作業範囲
- 危険度
- 止まる条件

依頼が曖昧すぎて、正直な合否判定を持つ Goal をまだ作れない場合は、推測で埋めずにヒアリングへ入ります。前の回答で次の質問が変わる時は、重要な質問を一つずつ出し、おすすめ回答も添えます。互いに影響しない不足事項は一度にまとめます。Goal を安全に定義できるようになったら、質問を続けるための質問はしません。コード、ドキュメント、資料を読めば分かることは、質問する前に確認します。

## Codex

`create_thread` は、ユーザーが別タスク化を明示した場合だけ使います。作業が分けられそうという理由だけでは使いません。複数の書き込みタスクへ並行展開する場合は、さらに次がすべて成り立つ必要があります。

- 独立した書き込み作業が2つ以上ある
- それぞれの担当範囲と確認方法がはっきりしている
- 共有部分をどうつなぐかが分かっている
- 別スレッド化で増える確認・統合作業より、短縮できる時間の方が大きい
- 既に Git の作業場所を分けられる状態になっている

並列化のためだけに Git を初期化したり、土台を作ったり、共有部分を設計したりしません。作業場所がまだ向いていない場合は、黙って別タスクを作ったり直列へ切り替えたりせず、足りない条件と必要な判断を報告します。ユーザーが単一の引き継ぎ先や新しいタスクを明示した場合は、複数タスク用の条件を一律には当てません。

Goal の実行中、`spawn_agent` は、フィードバックループ、並行探索、コンテキスト分離、または別の読み取り確認で完了判断が変わり得る時に使います。ここは能力説明ではなく、実行中の Codex に対するルールです。条件に当たるなら実際に subagent を起動し、「subagent を使える」という説明や親だけの自己確認で置き換えません。たとえば、調査ログを親のコンテキストから分ける、独立した論点を並行調査する、新しい証拠を別の focused pass へ戻す、実際の出力を新しいコンテキストから崩しにいく、といった使い方です。これは固定された段階や役割ではありません。親が、その時点の証拠、依存関係、コストから、直列・並列・追加 wave のどれが有効かを選びます。低リスクで十分な自動確認がある作業では不要です。最初は最小限にし、結果をまとめた後で追加が Done を変え得る場合だけ次を呼びます。確認役は読み取り専用とし、親が統合、修正判断、最終判断を持ちます。

## Claude Code

Claude Code でも同じ考え方で動かせます。Goal は、何を分けて確認するかを書きます。細かい実行方法は Claude Code 側に任せます。

## Goal が扱うこと

大きめの作業では、goal-setter は次を考えます。

- 何を達成したいか
- 何ができたら完了か
- 何で確認するか
- どこまで触ってよいか
- 何を変えてはいけないか
- 既に使われている表、資料、管理ファイルを増やさず更新するか
- どんな時に質問するか
- どこから読むか
- 必要な確認を弱めずに済ませるルール
- 長い作業で未確認一覧を更新しながら進むこと
- 止まる条件
- 完了前にどの強さで見直すか

短い作業には短い Goal を書きます。結果に影響しない内容は入れません。
