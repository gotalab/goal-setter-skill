# Runtime Behavior

goal-setter は goal text を書くスキルです。新しい scheduler を実装しているわけではなく、各 runtime に既にある goal / 並列化の仕組みを使います。

## Activation

| Runtime | 動き |
| --- | --- |
| Codex、worker tool 起動が不要 | native goal tool で goal を set できる |
| Codex、`spawn_agent` または `create_thread` が必要 | ユーザーが送る `/goal ...` 行を返す |
| Claude Code | ユーザーが送る `/goal ...` 行を返す |

Codex の worker tool 起動だけは特別です。`spawn_agent` と `create_thread` は明示的なユーザー依頼から発火するため、goal-setter が `/goal ...` の一行を渡し、ユーザーがそれを送ることで tool 起動を authorize します。これは `spawn_agent` を Done 前の最終検証だけに使う場合も同じです。

## 並列作業

成果が独立して個別検証できる単位に割れ、分ける手間より得が大きい場合、goal には次を入れます。

- 単位を発見するルール
- 単位ごとの担当範囲と証拠
- item-by-item の進捗
- 親 thread の統合チェック
- Done 前の確認レベル。リスクが高い主張に具体的な確認対象がある場合は、崩しにいく検証も含める

分割可否は、ファイル配置より先に、挙動の結合度、共有状態、統合リスクで判断します。ファイルパスは repo を読んだ後の手がかりであって、最初の境界ではありません。ユーザーが数を指定していない限り、subagent 数は固定しません。親 agent が独立性、リスク、コスト、統合できる証拠量を見て数と波を決め、各波を統合してから追加するか判断します。

## Codex

`create_thread` は、次がすべて成り立つ時だけ使います。

- 独立した書き込み単位が2つ以上ある
- それぞれの担当範囲と確認方法が安定している
- 共有部分をどうつなぐかが十分に分かっている
- 別スレッド化で増える確認・統合作業より、短縮できる時間の方が大きい
- 既に git/worktree が使える

並列作業のためだけに git 初期化、土台作り、共有部分の設計をしません。作業場所がまだ向いていない場合は serial に進めるか、repo 構造を変える前に確認します。

`spawn_agent` は、別の読み取り確認で Done 判断が変わり得る時に使います。調査、複数観点レビュー、崩しにいくレビュー、最終検証、ログ解析、既存挙動の把握などです。低リスクで強い自動確認がある作業では不要です。subagent は証拠、反証、不確実性、抜け、読み取り結果を返し、親 agent が統合、書き込み判断、最終判断を持ちます。

## Claude Code

Claude Code では、同じ分解構造を dynamic workflow として実行できます。goal は構造と「並列で fan-out せよ」という意図を書きますが、機構の細部は run に委ねます。読み取り段階は subagent、書き込み段階は worktree isolation になることがあります。

## Goal がカバーするもの

非自明な作業では、goal-setter は次を考慮します。

- 目的と、その成果がなぜ必要か
- objective と Done 条件
- 証跡と検証方法
- 目的、証拠、制約、範囲、次の試し方、停止条件に寄せる圧縮
- 表、レポート、ドキュメント、ダッシュボード、管理ファイルの既存正本を増やさず更新する扱い
- read-first anchors
- hard boundaries と、必要なチェックを弱めないルール
- 長い run の進捗ルール
- stop conditions
- リスクに合う最終確認。高リスクな主張に具体的な対象がある場合は、崩しにいく検証
- final report
- 不確実な調査では、棄却条件と停止条件を含む問いと仮説のループ

短いタスクには短い goal を書きます。結果に影響しない条項は入れません。
